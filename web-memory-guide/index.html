Claude on the Web
How Memory Actually Works

Why Claude sometimes forgets things you told it, why it sometimes mixes up details from other conversations, and how to set up the web app so it stops happening.

Mental model

Three layers, not one

When people say "Claude's memory" on the web, they usually mean one thing. It's actually three different systems stacked on top of each other, and they behave very differently. Knowing which layer a fact lives in is the single biggest predictor of whether Claude will get it right next time.

1 · Project Instructions

Always loaded · Authoritative

A custom prompt attached to a Project. Every conversation in that Project starts with this text loaded verbatim. If it's written here, Claude sees it on turn one, every time. This is where rules, preferences, and non-negotiable constraints belong.

2 · Project Knowledge

On demand · Reliable

Files you upload to a Project (docs, specs, schemas, style guides, transcripts). Claude searches them when a question touches their content. Canonical reference material belongs here — not in a past chat and not in memory.

3 · Memory (auto-captured)

Retrieved · Lossy

Compressed summaries of past conversations that Claude can surface in later chats. Useful, but never a source of truth. Memory is retrieved by similarity, summarised aggressively, and can conflate details from different conversations. This is the layer people think of as "Claude's memory" — and it's the one that forgets and mixes things up.

The rule: if a fact matters, promote it up the stack. Memory → Knowledge → Instructions. Anything critical goes in Instructions. Anything stable goes in Knowledge. Memory is a convenience, not a contract.
Symptom → cause

Why Claude "forgets"

Forgetting on the web almost never means the information was deleted. It means one of four very specific things happened. Each has a different fix.

"I told it this yesterday and today it has no idea."

Cause · Conversation scoping

You told it in a different conversation. By default, two chats don't share anything unless they live in the same Project and memory has captured the detail. A brand-new chat is a brand-new context — the previous one may as well not exist.

Fix: work inside Projects, and put anything you want to survive across chats into Instructions or Knowledge — not into the chat itself.

"It remembered the gist but got the specifics wrong."

Cause · Lossy summarisation

Memory isn't a transcript. It's a compressed summary — Claude keeps the shape of what you discussed, not the exact numbers, names, or code. The longer ago the conversation, and the longer that conversation was, the more detail has been squeezed out.

Fix: don't rely on memory for exact values. For anything precise — API keys, version numbers, file paths, quoted text — put it in Knowledge as a file, or state it again at the start of the chat.

"I set a preference weeks ago. It's ignoring it now."

Cause · Retrieval miss

Memory is searched, not loaded. If the current conversation doesn't look semantically similar to the chat where you set that preference, retrieval can miss it entirely. The memory is still there — Claude just didn't pull it this time.

Fix: preferences are instructions, not memory. Put them in Project Instructions so they load every turn regardless of what the conversation is about.

"The long chat started losing earlier details."

Cause · Context compression

Inside a single long conversation, older turns get summarised as the chat grows. This is separate from cross-chat memory — it's the model making room for new content by compressing what came before. Early instructions can soften or drift.

Fix: for long work, anchor the rules in Instructions (not the first message of the chat), and periodically re-state the constraints when you notice drift. Or start a fresh chat for a new phase.

Symptom → cause

Why Claude "mixes things up"

Mixing is more confusing than forgetting because Claude sounds confident while being wrong. It's almost always one of these:

"It's confusing two of my projects."

Cause · Cross-project bleed

If two different efforts share keywords — "the auth rewrite", "the React dashboard", "the API client" — a memory fragment from one can surface in the other. Retrieval goes by similarity, not by which project you meant.

Fix: use Projects as hard walls. One Project per distinct effort, with its own Instructions and Knowledge. If two projects are similar, make them more distinct in Instructions ("this is the internal dashboard, not the customer one"), not less.

"It combined two separate decisions into one thing that never happened."

Cause · Summary conflation

When past conversations get summarised, related-but-distinct decisions can get fused. "We chose Postgres" and "we chose pgvector for search" can collapse into "we chose Postgres with pgvector" even if they were separate calls made weeks apart.

Fix: anything important enough to get wrong belongs in Knowledge as a file you control — a decisions.md, an ADR, a spec. Don't leave high-stakes facts to be reconstructed from memory.

"It's using an old version of a decision we already changed."

Cause · Stale memory

Memories don't automatically update when you change your mind. Both versions can coexist, and retrieval may surface the older one. Nothing is wrong with Claude — it's just doing what memory is designed to do.

Fix: tell Claude explicitly: "forget what you know about X, the current state is Y." Or open the memory settings in the web app and delete the stale entry directly. Then update Instructions/Knowledge so the new state is authoritative.

The bigger point: if you're noticing "mixing" often, it means you're using memory as a store of record. That's not what it's for. Memory is a nudge — Instructions and Knowledge are the record.
Practical setup

How to make this stop happening

Most "Claude is getting worse" complaints on the web come down to the same structural fix: stop trusting memory for things memory was never meant to hold, and move those things into the layers that are reliable.

  • One Project per distinct effort. Don't pile unrelated work into one catch-all Project — that's the main source of cross-project bleed.
  • Write real Project Instructions. Role, preferences, style, hard rules, terminology. Treat it like a system prompt, because that's what it is.
  • Upload canonical docs to Knowledge. Specs, schemas, decision logs, style guides. Anything you'd be annoyed to re-explain.
  • Keep a decisions.md in Knowledge. Append new decisions with dates. When something changes, edit the file — don't hope memory caught it.
  • Don't let a single chat sprawl. When you switch tasks, start a new conversation. Long chats compress old turns and drift.
  • Audit memory occasionally. Open memory settings in the web app, skim what's there, delete anything stale or wrong.
  • Say "forget that" out loud. When you spot a wrong memory in the moment, correct it explicitly — Claude can remove or overwrite memory entries on request.
  • Say "remember that" out loud. Memory auto-captures aggressively but not perfectly. If a fact matters, name it so it's definitely captured.
One-liner test: if losing the memory system tomorrow would break your workflow, you're over-relying on it. The Project Instructions + Knowledge should carry the weight. Memory is the cherry on top.
Comparison

Web memory vs Claude Code memory

If you also use Claude Code (the CLI), its local memory system is a completely different thing — and worth knowing, because the mental models don't transfer.

Claude.ai (Web) Claude Code (CLI)
Storage Server-side, managed by Anthropic Local Markdown files on your machine
What you can see Memory settings panel — edit / delete entries Full file contents in your config directory
Scope Per-Project (and partly per-account) Per-project working directory
How it's loaded Retrieved by similarity (search) Index file loaded every session
Failure mode Forgets, mixes, conflates Gets stale — but you can read and fix it
Canonical facts go in Project Instructions + Knowledge CLAUDE.md + memory files

Both systems share the same principle: don't rely on implicit memory for things that matter. On the web, promote facts into Instructions or Knowledge. In Code, put them in CLAUDE.md or a named memory file. In both cases, the fix for "Claude keeps forgetting" is to stop asking memory to do a job it isn't built for.
