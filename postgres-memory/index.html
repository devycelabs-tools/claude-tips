Persistent Memory
with Postgres + MCP

Give Claude searchable, tagged, semantic memory that survives indefinitely — wired into every session automatically via hooks.

This guide builds on the Memory Guide — if you haven't set up local file memory yet, start there first. This is the next step up.

Overview

What this adds over file memory

The Memory Guide covers local Markdown files — zero infra, works out of the box. This guide adds a second tier on top: a Postgres-backed MCP server that gives Claude real search, automatic session hooks, and context continuity across compaction.

Tier 1 · Already covered

Local file memory

Typed .md files in ~/.claude/projects/. MEMORY.md index auto-loads each session. No infrastructure needed.

Limitation: No search. Claude reads files on demand but can't query across them. No session hooks. No context-compaction recovery.

Tier 2 · This guide

Postgres MCP memory

A Python MCP server backed by PostgreSQL. Tag filtering, full-text search, vector semantic search, session hooks, and checkpoint/restore across context compaction.

Requires: A server running Docker (homelab, VPS, or local machine).

Don't have a server? Tier 1 alone gives you 80% of the value. Come back to this guide when you have infrastructure and want search + hooks.

Session Lifecycle

What happens in a session

Three hooks wire memory into the session lifecycle automatically. You don't have to ask Claude to recall anything — it just happens.

Hook 1 · SessionStart

Surface pending work at the start

The SessionStart hook fires before Claude sees your first message. It injects a system-reminder telling Claude to call memory_search with specific tags before responding.

You choose the tags. Common pattern: pending-review for things needing attention, backlog for parked work, todo for open tasks. Any memory saved with those tags surfaces automatically at the top of the next session.

Hook 2 · PreCompact

Save working context before compression

When Claude Code compresses the conversation context (to free the context window), Claude loses its short-term working memory. The PreCompact hook fires just before that happens and calls checkpoint_save.

The checkpoint captures: what Claude is working on, decisions made this session, critical facts, and unresolved items.

Hook 3 · PostCompact

Restore working context after compression

The PostCompact hook fires after compression. It calls checkpoint_restore, fetches the last saved checkpoint, and injects it as a system-reminder. Claude resumes with reconstructed working state.

This is called Hippocampal Replay — named after the neuroscience mechanism where the brain re-activates short-term memories during sleep to transfer them to long-term storage.

── A TYPICAL SESSION ────────────────────────────────────────────────── Session opens MEMORY.md auto-loads (always-on) SessionStart hook → memory_search(pending-review, backlog, todo) Claude surfaces any items before responding to you You work with Claude Claude calls memory_search() when it needs to recall something Claude calls kv_set/kv_get for ephemeral tool state On accepted decisions → auto-saves via memory_save() Context gets long → compaction triggered PreCompact hook → checkpoint_save(working_on, decisions, facts, unresolved) Context compressed → short-term memory gone Claude resumes PostCompact hook → checkpoint_restore() Working context injected as system-reminder Claude continues as if nothing happened
Data Model

What gets stored in Postgres

The memory table

The primary memory store. Every call to memory_save writes here. Searchable by tag array (GIN index) and full-text (tsvector trigger).

ColumnTypePurpose
idBIGSERIALPrimary key — reference this to fetch or delete a specific memory
tsTIMESTAMPTZWhen saved (auto)
tagsTEXT[]Array of tags — GIN indexed for fast @> containment queries. All tags in your query must match.
summaryTEXTShort one-line description. Used for FTS and for display in search results.
detailTEXTOptional long-form content — the why, the context, the constraints.
sourceTEXTWho saved it: claude, user, hook
search_vectortsvectorFTS index maintained by trigger from summary + detail. Never write to this directly.
Write summaries with natural search vocabulary. FTS is literal until you have vector search enabled — it finds words, not concepts. If you search for "database timeout" later, the memory needs to contain those words. Avoid jargon or abbreviations in summaries.

The knowledge_chunks table (semantic / RAG search)

When a memory is saved, it is also embedded into knowledge_chunks via a local embedding model. This powers rag_search — cosine similarity search via pgvector's <=> operator.

ColumnPurpose
contentThe durable asset — text that survives any model change. Never store only the vector.
embeddingvector(N) — rebuildable cache. If you swap embedding models, re-ingest from content.
embed_modelWhich model produced this vector. Lets you filter or selectively re-embed after a model upgrade.
tieractive | archive — demote old chunks, never delete. Keeps history while focusing search on current data.
sanitize_statuspassed | quarantined | cleared — security gate. Only passed/cleared chunks are returned by rag_search.
chunk_typecode | config | docs | synthesis | scribe — filter searches by type
file_path / repoProvenance — where this chunk came from. Memories use data-mcp/memory/{id} as a pointer back to the memory row.

Other tables

checkpoints

Stores the structured working-state snapshot written by PreCompact and read by PostCompact. Fields: working_on, decisions[], critical_facts[], unresolved[].

Always fetches the most recent checkpoint. Only the last one matters.

kv_store

Simple key-value JSON store for ephemeral state that needs to outlive a single tool call but isn't worth a full memory. Job IDs, flags, tool state, UI cursors.

Not for memories — use the memory table for anything that should outlive a session.

Vector Search

How semantic search works

Tags and full-text search find what you asked for. Vector search finds what you meant. Here's the full pipeline, end to end.

The model — voyage-4-nano

voyage-4-nano is Voyage AI's smallest embedding model, designed for retrieval. It runs locally as a containerised HTTP microservice — no API key, no external calls, fully offline. The variant used here is a quantized GGUF build (q8_0) with a linear projection layer that outputs 2048-dimensional vectors.

On a budget homelab CPU (i5-10500T) it embeds at roughly 2–3 chunks per second. That's slow enough to notice on a bulk ingest, but fast enough that a single memory_save or rag_search call completes without blocking. The MCP server calls it over HTTP and degrades gracefully if it's unreachable — memories are saved without embeddings and can be ingested in a later batch run.

The embedder API

One endpoint is all the MCP server needs:

POST /embed Content-Type: application/json { "text": "the text to embed" } → { "ok": true, "embeddings": [[0.021, -0.134, ...]] } // 2048 floats GET /healthz → { "ok": true, "model": "voyage-4-nano-q8_0+linear", "dim": 2048 }

On memory_save, the MCP server concatenates the summary, detail, and tags into one string and sends it to /embed. The returned vector is stored alongside the text in knowledge_chunks. On rag_search, the query string is embedded the same way and the resulting vector is used for similarity search.

Storage — pgvector + IVFFlat index

Vectors are stored in the knowledge_chunks table as a vector(2048) column. pgvector adds an IVFFlat index (vector_cosine_ops) that makes approximate nearest-neighbour search fast at scale.

IVFFlat needs rows to build. The index creation is deferred until the table has data — building it on an empty table produces a useless index. The schema migration script checks row count and skips index creation if the table is empty; run it again after your first bulk ingest.

Search — cosine similarity via the <=> operator

pgvector's <=> operator computes cosine distance (0 = identical, 2 = opposite). The query converts it to similarity with 1 - (embedding <=> query_vec), then orders ascending by distance so the closest chunks come first:

SELECT content, chunk_type, 1 - (embedding <=> $query_vec) AS similarity FROM knowledge_chunks WHERE sanitize_status IN ('passed', 'cleared') ORDER BY embedding <=> $query_vec -- ascending distance = descending similarity LIMIT 5;

The rag_search tool accepts a threshold parameter (0.0–1.0) to filter out low-confidence results. In practice a threshold of 0.5–0.6 removes noise without losing genuinely relevant matches.

Bulk ingest for existing memories

New memories are auto-embedded on save. For memories that existed before the embedder was set up, an ingest script does a catch-up pass — it reads every row in the memory table, checks whether a corresponding knowledge_chunks row already exists (by file_path = 'data-mcp/memory/{id}'), skips those, and embeds the rest. Safe to re-run at any time.

Content is the durable asset. Vectors are disposable cache. If you swap embedding models or migrate databases, don't move the vectors — re-run the ingest script against the new model. The text in content always survives; the numbers in embedding are just an index over it.
MCP Tools

Tools Claude gets

The data-mcp server exposes these as Claude Code MCP tool calls. Claude can call them in any session where the MCP is registered.

ToolWhat it does
memory_saveSave a memory. Requires summary + optional tags[] and detail. Auto-embeds into knowledge_chunks on write.
memory_searchSearch by tags AND/OR full-text. Tag query uses GIN @> — all supplied tags must be present. FTS uses plainto_tsquery with ts_rank ordering.
memory_getFetch one memory by ID — useful when the index summary isn't enough and you need the full detail.
memory_deleteRemove a memory permanently by ID.
rag_searchSemantic vector search. Queries the local embedder → cosine similarity via pgvector. Use when keyword search misses the mark ("find something about auth timeouts" vs. exact phrase match).
kv_set / kv_get / kv_list / kv_deleteKey-value store. JSON values, upsert on conflict.
checkpoint_saveWrite a session checkpoint. Called by PreCompact hook. Pass structured fields, not a free-form dump.
checkpoint_restoreFetch the latest checkpoint. Called by PostCompact hook.
log_action / get_logsAudit trail — what Claude did, when, to what, with what result. Useful for reviewing what happened in past sessions.
memory_search vs rag_search: Use memory_search for categorical recall (give me all backlog items) and known-phrase lookup. Use rag_search for fuzzy "find something like X" — it matches meaning, not words. In practice, start with tags, fall back to FTS, then semantic.
CLAUDE.md Rules

When Claude should auto-save

Memory saves should be automatic — you shouldn't need to ask each time. Define these rules in your CLAUDE.md:

Save without asking when:

  • A new service, component, or dependency is proposed and accepted
  • An architectural direction is chosen over alternatives — the "we went with X not Y" moment
  • A non-obvious technical constraint or gotcha is discovered
  • You explicitly say "remember that…"

Do NOT save:

  • In-progress work or temporary state — use checkpoints for that
  • Obvious choices where no alternatives were considered
  • Things already in CLAUDE.md or the existing memory files (check first)
  • Code patterns — those are derivable by reading the repo
  • Ephemeral task details that only matter this session
The "why" is what makes memories useful. A memory that says "use Postgres" is useless. A memory that says "use Postgres — we switched from SQLite after the migration failed under concurrent writes" tells Claude why and lets it apply the principle to edge cases.

The feedback and project memory types are especially important here — they preserve the reasoning behind decisions, not just the outcome. See the Memory Guide for memory type details.

Recall Strategy

How Claude should look for context

Define this order in your CLAUDE.md. It prevents Claude from going straight to slow or expensive lookups when faster ones would work:

1. CLAUDE.md always in context — check here first, never redundantly recall 2. MEMORY.md index auto-loaded — scan for relevant files, then fetch on demand 3. memory_search() tag or FTS query when the index isn't enough 4. rag_search() semantic search for fuzzy "find something like X" queries 5. Ask the user last resort — don't scan local files unprompted
Stack

What you need to run this

PostgreSQL + pgvector

Docker image: pgvector/pgvector:pg16

Needs CREATE EXTENSION vector after first start. The memory and knowledge_chunks tables are created by the MCP server on first boot.

data-mcp server

Python FastMCP, stdio transport. One Python file + a schema migration file. Registered in Claude Code via claude mcp add -s user.

Transport is stdio — runs as a subprocess on your desktop, talks to Postgres over the network.

Local embedder (optional)

A small HTTP microservice that wraps a local embedding model and exposes a single POST /embed endpoint. The MCP server calls it on every memory_save and rag_search.

The setup described in this guide uses voyage-4-nano running fully offline — no API key, no external calls. At ~2.75 chunks/sec on a budget homelab CPU it's slow enough that you notice it, but fast enough that it never blocks anything. See the Vector Search section below for the full pipeline.

Claude Code hooks

Configured in settings.json. SessionStart, PreCompact, PostCompact. Each hook is a shell command that prints a system-reminder to stdout — Claude Code injects that into the session.

Minimum footprint: Postgres + pgvector container + the data-mcp Python file. Skip the embedder if you don't need semantic search — memory_search with tags and FTS covers most use cases without it.
Limitations

Known limitations and design notes

FTS is literal until you add pgvector

plainto_tsquery does stemming but is still keyword-based. Searching "database caching" won't find a memory about "connection pooling". Use tags for categorical recall; use rag_search for fuzzy semantic queries. Write summaries with natural search vocabulary.

MEMORY.md truncates at ~200 lines

It's always-on context — keep it lean. One line per memory, ~150 chars max. If it grows beyond that, the oldest entries get truncated out of the context window and become invisible to Claude without an explicit search.

No built-in deduplication

If you save the same fact twice with different phrasing, you get two records. The CLAUDE.md rule to "check before saving" is soft — not enforced by the database. Over time this creates clutter. Periodically review and delete stale or duplicate memories via memory_delete.

Embeddings are a cache, not the source of truth

If you migrate databases or swap embedding models, don't migrate vectors — re-ingest from the content column. The ingest script reads all memory rows and re-embeds anything not already in knowledge_chunks. Content is the durable asset; vectors are disposable.

Memory records can become stale

A memory that cites a file path or function name was accurate when saved. File paths get renamed. Before acting on a recalled memory, verify against the current codebase. If a recalled memory conflicts with what you observe now, trust current state — and update or delete the stale memory.
