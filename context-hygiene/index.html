Context Hygiene

Practical habits that keep your context window from filling up too fast — so Claude stays sharp for longer, in every session.

The Problem One-Time Setup Session Habits Prompt Hygiene When You're Filling Up Quick Reference
Background

Why Context Fills Up

Every message, tool result, file read, and code block in a conversation takes up context. Claude Code sessions can run for hours — that accumulated history eventually crowds out space for new work.

When context is nearly full, Claude may start losing earlier parts of the conversation, produce less coherent responses, or get cut off mid-task. The usual causes:

1

Large file reads

Asking Claude to read whole files — especially large ones — dumps all that content into the window at once.

2

Long unbroken sessions

Multi-hour conversations accumulate every back-and-forth exchange, even the exploratory ones that led nowhere.

3

Repeated re-explanation

Re-stating project background, preferences, and constraints that Claude already knows wastes space every session.

4

Broad tool calls

Searches and reads that return hundreds of lines of results — when you only needed a small slice of them.

Context window usage ⬆ bad habits accelerate this
0–60% — full power
60–85% — Claude starts summarising
85%+ — degraded, act now

The good news: most context burn is preventable with a handful of habits. None of them are complicated — they just need to become automatic.

One-Time Setup

Put Context Where It Lasts

The highest-leverage move is getting stable context out of the conversation window and into persistent storage — where Claude loads it efficiently rather than you re-stating it every session.

Setup
1

Write a CLAUDE.md for your project

A CLAUDE.md file at the root of your repo is automatically loaded at session start. Put everything Claude needs to know about the project there: tech stack, conventions, which directories to avoid, recurring commands.

This replaces the "let me explain the context" paragraph you type at the start of every session. Write it once, update it when things change.

CLAUDE.md — what to include## Stack Node 20 + TypeScript. Tests with Vitest. No Jest. ## Conventions - Prefer named exports. No default exports except pages. - snake_case for DB columns, camelCase everywhere else. ## Don't touch - src/legacy/ — read-only, being sunset by another team ## Run commands - Tests: npm test - Dev server: npm run dev (port 3000)
Setup
2

Use memory for cross-session facts

Claude Code's memory system (see the memory guide) stores non-obvious facts that don't belong in the codebase: your preferences, architectural decisions, project constraints, and "we chose X because Y" context.

Without memory, Claude re-derives this context from scratch every session — burning window space in the process. With memory, it loads a lean summary and reads details only when needed.

Key insight: Memory isn't just about Claude remembering things. It's about keeping that information out of the active context window until it's actually needed — which is most of the time.
Setup
3

Install only the MCPs you actually use

Every active MCP server adds tool definitions to your context window — even if you never call them. A bloated MCP list silently eats into available space before you've typed a single message.

Audit your MCP config periodically. Disable servers you haven't used in weeks. Prefer focused MCPs over general-purpose ones that expose 30+ tools you'll never need.

During Each Session

Keep Sessions Focused

Context fills fastest when sessions sprawl. Staying focused on one concern per session is the single biggest behavioural change you can make.

Session
4

One task per session

When a session spans multiple unrelated tasks, context from task A is still present when you start task B — and task C — even though it's no longer useful. Each context switch compounds the waste.

Finish a task, use /clear to wipe the slate, then start the next one fresh. The cleared session doesn't lose any work — it loses the conversation history you no longer need.

command/clear
Session
5

Cut off exploratory dead ends early

When an approach isn't working and you pivot, the failed exploration stays in context even after you've moved on. It confuses future responses and burns space.

If a long exploration goes nowhere, /clear and restart with a tighter prompt. You don't lose the insight — you gained that from reading the output. You just stop carrying the dead weight.

Session
6

Break large features across sessions

Implementing a large feature in one marathon session accumulates design discussions, implementation attempts, and back-and-forth into one giant window. By the end, Claude may be working with a compressed view of what happened at the start.

Instead: design in one session, implement in another. Commit progress between sessions. A CLAUDE.md or story file carries the handoff context efficiently.

Session
7

Use subagents for isolated, heavy work

When Claude uses the Agent tool to spawn a subagent (e.g. to explore a codebase, run tests, or do research), the subagent's full read/search output is summarised before it comes back — not dumped verbatim into your window.

For tasks that involve reading lots of files or doing broad searches, explicitly asking Claude to delegate to a subagent keeps the detail out of the main window. The main conversation only sees the conclusion.

exampleUse a subagent to find all usages of the old auth pattern across the codebase and summarise them — I don't need the raw results.
How You Write Prompts

Prompt Hygiene

How you phrase requests directly affects how much context each exchange consumes. Vague prompts generate more back-and-forth. Overly broad requests pull in more data than you need.

Prompt
8

Be specific — file + line beats vague description

A vague prompt creates a diagnostic loop: Claude asks clarifying questions, reads around the problem area, makes a wrong guess, you correct it. Each exchange costs context.

A specific prompt goes straight to implementation. The difference in context cost can be 3–5x.

vague (expensive)Something seems off with the login flow.
specific (cheap)Fix the validation logic in src/auth/login.ts at line 84 — it's rejecting valid email addresses that contain a + character.
Prompt
9

Don't paste files — let Claude read them

Pasting a file into the chat adds its full content to context immediately. Claude reading it with the Read tool does the same thing — but only when it actually needs it, and only the relevant portion.

Never paste large files into the conversation. If you want Claude to look at a file, say the path and let it read it on demand.

Watch out for: Asking Claude to "take a look at everything in src/" — that's an implicit instruction to read potentially dozens of files upfront. Prefer targeted asks: "Look at src/auth/login.ts and src/auth/session.ts."
Prompt
10

Don't repeat context Claude already has

If your project context is in CLAUDE.md and your preferences are in memory, you don't need to restate them. Prepending every message with "as I mentioned, this is a TypeScript project using Vitest…" adds tokens without adding information.

Trust the loaded context. If Claude seems to have forgotten something, ask it to re-read the relevant file — don't repeat it inline.

Prompt
11

Ask for targeted searches, not full scans

Prompts like "search the whole codebase for X" can return hundreds of results. You usually only needed to know if X exists and where. Ask for the specific thing you need:

instead ofSearch the whole codebase for any reference to the old payment API.
preferFind all files under src/ that import from @acme/payments-v1 — just the file names.
When You're Running Low

Recovery Tactics

Even with good habits, long sessions fill up. Here's how to handle it without losing your work.

Recovery
12

Use /compact to summarise history

Claude Code's /compact command replaces the full conversation history with a compressed summary, freeing up substantial space while keeping Claude oriented on what you're doing.

Run it proactively — before you hit the wall — rather than waiting until Claude is already degraded. A good time: after finishing a discrete chunk of work but before starting the next phase.

command/compact
Recovery
13

Ask for a summary before you /clear

Before clearing a session with important in-progress context, ask Claude to summarise the current state — decisions made, what changed, and what comes next. Even a short paragraph pasted into a note is enough to re-orient the next session quickly.

This takes 10 seconds and costs almost nothing. Skipping it and trying to reconstruct context from scratch costs much more.

promptBefore I clear — summarise where we are: what the goal is, what we've done, and what the immediate next step is.
Recovery
14

Write a handoff note to the next session

If you're mid-task and running out of context, ask Claude to write a concise handoff: what the goal is, what's been done, what the next step is, and any gotchas discovered. Save it to a file (.claude/stories/current.md is a good convention).

Then /clear and start fresh with "read .claude/stories/current.md and continue." The new session starts oriented without re-reading the whole history.

prompt to generate a handoffWrite a concise handoff note to .claude/stories/current.md covering: the goal, what we've done so far, the next step, and any gotchas I should know.
