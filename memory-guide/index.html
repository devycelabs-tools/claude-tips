Claude + Local Memory
A Practical Guide

How Claude remembers context across sessions, what it stores, and how you can ask it to show its own working.

Concept

What is the Memory System?

By default, every Claude Code session starts fresh — no memory of past conversations. The local memory system changes that. Claude writes facts it learns about you, your project, your preferences, and past decisions to a set of Markdown files in your config directory.

These files are loaded at the start of future sessions, giving Claude persistent context without you having to re-explain everything.

💬 Session You work with Claude
→
🧠 Claude Learns Non-obvious facts emerge
→
💾 Memory Saved Written to .md files
→
🔁 Next Session Context pre-loaded

Where the files live

Memory files are stored in your Claude config directory, under a project-scoped path:

~/.claude/projects/<your-project-path>/memory/ MEMORY.md ← index of all memories (loaded every session) user_role.md ← who you are, your expertise feedback_testing.md ← how Claude should approach your work project_auth.md ← current project decisions + context reference_apis.md ← pointers to external resources

MEMORY.md is an index — one line per memory file. Claude reads it at session start to know what context is available, then reads individual files when they're relevant to the current task.

Four types of memory

User

Who you are

Your role, expertise, background. Helps Claude tailor explanations to your level and avoid over-explaining things you already know.

"Senior Go engineer, new to this project's React frontend"

Feedback

How to work with you

Corrections and confirmed approaches. Prevents Claude from repeating the same mistake across sessions.

"Don't mock the DB — we got burned when mocks diverged from prod"

Project

What's happening

Ongoing initiatives, deadlines, architectural decisions, and the "why we chose X over Y" context.

"Auth rewrite is compliance-driven, not tech debt"

Reference

Where to look

Pointers to external systems: Linear boards, Grafana dashboards, Slack channels, internal wikis.

"Pipeline bugs tracked in Linear project INGEST"

What Claude won't save: Code patterns, git history, file contents, or anything derivable by reading the repo. Memory is for the non-obvious stuff — preferences, constraints, decisions, and context that doesn't live in the code itself.
How-to

Asking Claude About Its Own Memory

Claude can read its own memory files and tell you what it knows, what decisions it has recorded, and how it has been instructed to behave. You just have to ask.

There's no special command syntax. These are plain English prompts you type at the start of a session.

Tip: Ask at the start of important sessions — before Claude does anything. It takes 10 seconds and catches stale context before Claude acts on it.
Prompt What it surfaces
What do you remember about me? Your profile, expertise, and preferences Claude has saved
What feedback have you recorded about how to work with me? Corrections and confirmed approaches from past sessions
What do you know about this project? Current project context, architectural decisions, ongoing work
Read your MEMORY.md and summarise everything you know Full memory index — all topics Claude has context on
What decisions have we made that I should know about? Architectural choices, "we chose X over Y" moments
Is any of your memory about this project outdated? Claude cross-checks memory against current code + git
Forget everything you know about X Claude deletes or updates a specific memory file
Remember that [fact] Explicitly adds a memory Claude wouldn't have captured on its own

Best Practices

Getting the Most Out of Local Memory

Start sessions with a memory check

Ask "what do you remember about me?" at the start of important sessions. It takes 10 seconds and catches stale context before Claude acts on it.

Explicitly save non-obvious facts

Claude saves things automatically, but it can't read your mind. If a decision, constraint, or preference matters to future work — say "remember that…" out loud. The prompt is the save button.

Correct mistakes immediately

If Claude does something wrong, say "don't do that — here's why." That correction gets saved as feedback and prevents the same mistake in future sessions. Corrections are more valuable than instructions given in advance.

Memory ≠ truth

Memories are snapshots in time. File paths get renamed, decisions get reversed, deadlines pass. Ask Claude to verify memory against the current codebase before acting on it for anything important.

The snap command

Before ending a long session or clearing context, say "snap" or "take a memory snapshot". Claude will extract the non-obvious things worth saving from the session — decisions, discoveries, constraints — and write them to memory files before you lose the context.
